## 2.软件工程
- 软件工程过程
    - P(Plan)—软件规格说明
    - D(Do)—软件开发
    - C(Check)—软件确认
    - A(Action)—软件演进

- 软件工程模型
    - 软件过程模型
        - 瀑布模型(Waterfall Model)
            - 特点: 线性(Linear), 严格串行(Strictly Sequential), 文档驱动(Documentation Driven), 因果关系紧密相连，前一个阶段的输出时下一个阶段的输
            - 优点: 简单直接(Simple and Direct)
            - 缺点: 难以适应变更(Requirements Change), 风险控制弱(Risk Control Weak
        - 原型化模型(Prototype Model), 又称快速原型法(Rapid Prototyping)
            - 特点: 原型化开发阶段和目标软件开发阶段两个阶段
            - 类型: 抛弃型(Throw-away) / 演化型(Evolutionary)
            - 试用场景: 需求模糊(Fuzzy Requirements)
        - 螺旋模型(Spiral Model)
            - 特点: 在快速原型法上扩展而而成，这种模型把整个软件开发的流程分成多个阶段，每个阶段都包括4个部分:
                - 1. 目标设定(Objective Setting)
                - 2. 风险分析(Risk Analysis)
                - 3. 开发和有效性验证(Development and Validation)
                - 4. 评审(Planning/Review)
            - 试用场景: 大型复杂系统(Large-scale Systems)
    - 敏捷模型或敏捷方法(Agile Modeling)
        - 特点：
            - 1. 适应性而非预测性
            - 2. 面向人而非过程
            - 3. 迭代增量
        - 各种敏捷方法：
            - XP(极限编程 - Extreme Programming)
                - 特点: 即是轻量、灵巧的，也是严谨、周密的开发方法
                - 价值观:
                    - 交流(Communication)
                    - 朴素(Simplicity)
                    - 反馈(Feedback)
                    - 勇气(Courage)
            - Scrum
                - 特点：侧重于项目管理，迭代式增量软件开发过程. 使用Backlog(待办列表)来管理产品需求, 并从中选择优先级高的需求组成Sprint Backlog(冲刺待办列表)
            - 水晶系列(Crystal Methodologies)
                - 特点：以人为中心的理念，提倡机动性
            - FDD(特征驱动开发 - Feature Driven Development)
                - 特点
                    - 迭代开发模型
                    - 强调软件开发3要素：人、过程、技术
                - 6个关键角色
                    - 项目经理
                    - 首席架构设计师
                    - 开发经理
                    - 主程序员
                    - 程序员
                    - 领域专家
                - 5个核心过程
                    - 开发整体对象模型
                    - 构造特征列表
                    - 计划特征开发
                    - 特征设计
                    - 特征构建
    - 统一过程(RUP: Rational Unified Process)
        - 特点：
            - 用例驱动(Use Case Driven)
                - 用例贯穿需求分析、设计、测试、实现
            - 以体系结构为中心(Architecture Centric)
            - 迭代和增量(Iterative & Incremental)
        - 生命周期，4个阶段：
            - 初始
            - 细化
            - 构造
            - 移交
        - 核心概念
            - 角色
            - 活动
            - 制品
            - 工作流
        - 9个核心工作流
            - 业务建模(Business Modeling)
            - 需求(Requirements)
            - 分析与设计(Analysis & Design)
            - 实现(Implementation)
            - 测试(Test)
            - 部署(Deployment)
            - 配置与变更管理(Configuration & Change Management)
            - 项目管理(Project Management)
            - 环境(Environment)
        - 4+1 视图
            - 逻辑视图(Logical View) - 最终用户(User)
            - 实现视图(Implementation View) - 程序员(Programmer)
            - 进程视图(Process View) - 系统集成人员(System Integrator)
            - 部署视图(Deployment View) - 系统工程师(System Engineer)
            - 用例视图(Use Case View) - 测试人员(Tester)
    - 基于构建的软件工程
    - 净室软件工程
    
- 软件能力成熟度模型CMMI(Capability Maturity Model Integration)
    - 5个成熟度等级
        - 初始级(Initial) - 混乱
            - 状态: 过程随意、无序
        - 已管理级(Managed) - 项目级
            - 状态：项目层面有基本的管理
        - 已定义级(Defined) - 组织级
            - 状态：全公司有统一的标准
        - 量化管理级(Quantitatively Managed) - 数据驱动
            - 状态：使用数据驱动管理
        - 优化级(Optimizing) - 持续改进
            - 状态：持续改进, 不断追求完美

- 需求工程

        - 功能需求(包括非功能需求)

                - 现场观察
                - 原型化方法
                - 头脑风暴法
            - 遵循的基本步骤
                - 开发高层的业务模型

- 系统分析与设计
    - 结构化方法(SASD: Structured Analysis and Design)
        - 特征： 面向功能(过程)，自顶向下，数据与行为分离, 适合需求明确、变化小的系统
        - 结构化分析(SA: Structured Analysis)
            - 分析手段(方法):
                - 数据流图(DFD: Data Flow Diagram)
                    - 四种基本元素
                        - 数据流(Data Flow)
                        - 处理(Processing)
                        - 数据存储(Data Store)
                        - 外部项(External Entity)
                    - 过程及步骤
                        - 明确目标、确定系统范围
                        - 建立顶层DFD图
                        - 构建第一层DFD分解图
                        - 开发DFD层次结构图
                        - 检查确认DFD图
                - 数据字典(DD: Data Dictionary)
                    - 组成
                        - 数据项(Data Item)
                        - 数据结构(Data Structure)
                        - 数据流(Data Flow)
                        - 数据存储(Data Store)
                        - 处理过程(process)
                - 结构化语言
                - 判定表
                - 判定树
            - 分析步骤:
                - (1)分析业务情况，做出反映当前物理模型的数据流图(Data Flow Diagram,DFD);
                - (2)推导出等价的逻辑模型的DFD;
                - (3)设计新的逻辑系统，生成数据字典和基元描述；
                - (4)建立人机接口，提出可供选择的目标系统物理模型的DFD;
                - (5)确定各种方案的成本和风险等级，据此对各种方案进行分析；
                - (6)选择一种方案；
                - (7)建立完整的需求规约。
        - 结构化设计(SD: Structured Design)
            - 阶段
                - 概要设计: 确定总体结构，模块功能和调用关系
                    - 工具: 系统结构图(SC: Structure Chart)或模块结构图
                - 详细设计: 每个模块的具体实现细节
                    - 工具:
                        - 图形工具
                            - 程序流程图(PFD: Program Flow Diagram) - 只描述过程不描述数据
                            - NS流程图或盒图(Nassi-Shneiderman Diagram) - 强制结构化，功能域明确，易于表示嵌套
                            - PAD图(Problem Analysis Diagram) - 取代程序流程图，允许递归使用
                        - 伪代码
            - 耦合: 模块之间的关系
                - 非直接耦合
                - 数据耦合
                - 标记耦合
                - 控制耦合
                - 通信耦合
                - 公共耦合
                - 内容耦合
            - 内聚: 模块内部的关系
                - 功能内聚
                - 顺序内聚
                - 通信内聚
                - 过程内聚
                - 时间内聚
                - 逻辑内聚
                - 偶然内聚
        - 结构化编程(SP: Structured Programming)
        - 数据库设计
    - 面向对象方法(OO: Object-Oriented)
        - 特征： 面向对象(数据+行为)，自底向上，封装(数据+行为)， 适合需求复杂、变化大的系统
        - 面向对象分析(OOA: Object-Oriented Analysis)
            - 基本步骤
                - 确定对象和类 - 找出系统系统里面的名称
                - 确定结构 - 确定对象之间的关系
                - 确定主题 - 定义事务的总体概貌
                - 确定属性 
                - 确定方法
        - 面向对象设计(OOD: Object-Oriented Design)
            - Class类的3种类型
                - 实体类(Entity Class)
                    - 特征：对应数据库表，映射显示世界的实体，保存需要永久存储的信息
                - 控制类(Control Class)
                    - 特征：控制用例工作的类，对应动词+名次A(AuthManager)
                - 边界类(Boundary Class)
                    - 特征：封装在用例类，处理流动的信息或数据流。 窗口、报表、接口，隔离外部环境的变化
        - 面向对象编程(OOP: Object-Oriented Programming)
            - 特点
                - 封装: 将数据(属性)和对数据的操作(行为或方法)组装在一起
                - 继承: 类与类之间的关系，子类继承父类的属性和方法
                - 多态: 同一个方法在不同的对象上表现出不同的行为
        - 数据持久化与数据库

- 软件测试
    - 测试方法
        - 动静
            - 静态测试(Static Testing) - 不运行程序
            - 动态测试(Dynamic Testing) - 运行程序
        - 内外
            - 黑盒测试(Black-box Testing) - 不看代码
            - 白盒测试(White-box Testing) - 看内部代码（逻辑结构）
            - 灰盒测试(Gray-box Testing) - 同时看输入输出和部分内部代码
        - 人机
            - 自动化测试(Automated Testing) - 把人驱动变成机器驱动
    - 测试阶段
        - 单元测试(Unit Testing)
        - 集成测试(Integration Testing)
        - 系统测试(System Testing)
        - 性能测试(Performance Testing)
        - 验收测试(Acceptance Testing)
        - 其他测试
            - AB测试(A/B Testing)
            - Web测试(Web Testing)
            - 链接测试(Link Testing)
            - 表单测试(Form Testing)
