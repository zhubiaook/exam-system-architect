# 第8章 系统质量属性与架构评估

- **8.1 软件系统质量属性**
  - **8.1.1 概念**
    - **6种维度特性 (GB/T 16260.1)**：功能性、可靠性、易用性、效率、维护性、可移植性
    - **开发期属性**：易理解性、可扩展性、可重用性、可测试性、可维护性、可移植性
    - **运行期属性**：性能、安全性、可伸缩性、互操作性、可靠性、可用性、鲁棒性
  - **8.1.2 面向架构评估的质量属性 (重点)**
    - **性能 (Performance)**：响应能力 (响应时间、吞吐量)
    - **可靠性 (Reliability)**：维持功能特性的能力 (MTTF, MTBF)
      - **容错** (Fault Tolerance)：出错时自我修复/正确行为
      - **健壮性** (Robustness)：出错时按定义方式终止/保护系统
    - **可用性 (Availability)**：正常运行时间比例 (故障间隔、恢复速度)
    - **安全性 (Security)**：阻止非授权使用 (机密性、完整性、不可否认性、可控性)
    - **可修改性 (Modifiability)**：变更成本
      - **可维护性** (修复缺陷)、**可扩展性** (增加特性)、**结构重组** (调整关系)、**可移植性** (环境适应)
    - **功能性 (Functionality)**：完成期望工作的能力
    - **可变性 (Variability)**：架构经扩充/变更成为新架构的能力 (产品线基础)
    - **互操作性 (Interoperability)**：与其他系统交互的能力
  - **8.1.3 质量属性场景描述**
    - **六要素**：刺激源 (Source)、刺激 (Stimulus)、环境 (Environment)、制品 (Artifact)、响应 (Response)、响应度量 (Measure)
    - **常见场景**：可用性、可修改性、性能、可测试性、易用性、安全性

- **8.2 系统架构评估**
  - **8.2.1 重要概念**
    - **敏感点 (Sensitivity Point)**：影响特定质量属性的架构特性
    - **权衡点 (Tradeoff Point)**：影响多个质量属性 (往往是冲突) 的架构特性
    - **风险承担者 (Stakeholders)**：客户、用户、架构师、开发人员、测试人员、经理等
  - **8.2.2 评估方法 (重点)**
    - **1. SAAM (基于场景的架构分析方法)**
      - **特点**：最早，主要关注 **可修改性**，也用于功能分析
      - **输入**：问题描述、需求声明、架构描述
      - **过程**：场景开发 -> 架构描述 -> 单个场景评估 -> 场景交互 -> 总体评估
    - **2. ATAM (架构权衡分析方法)**
      - **特点**：在SAAM基础上发展，关注 **多属性权衡** (性能、安全、可修改、可用)
      - **工具**：**效用树 (Utility Tree)** —— 质量属性分类与优先级 (重要度, 难易度)
      - **阶段**：
        - 1. **演示** (介绍ATAM/业务/架构)
        - 2. **调查和分析** (架构方法, 场景, 效用树, 分析)
        - 3. **测试** (头脑风暴优先场景, 分析)
        - 4. **报告** (展示结果：风险/敏感点/权衡点)
      - **产出**：风险、非风险、敏感点、权衡点
    - **3. CBAM (成本效益分析方法)**
      - **特点**：经济模型，**ROI (投资回报率)** 驱动决策
      - **步骤**：整理场景 -> 求精 -> 优先级 -> 分配效用 -> 策略映射 -> 计算收益 -> ROI排序选择
    - **4. 其他方法**
      - **SAEM** (外部/内部属性建模)
      - **SAABNet** (定性知识, 贝叶斯信念网络, AI)
      - **SACMM** (架构修改度量, 图内核距离)
      - **SASAM** (静态评估, 预期vs实际架构一致性)
      - **ALRRA** (可靠性风险, FMEA)
      - **AHP (层次分析法)**：多准则决策，定性转定量，权重计算

- **8.3 ATAM 评估实践 (案例：Hoover vs Banking)**
  - **案例背景**：Event框架的两种架构
    - **Hoover架构**：构件独立，Event Manager核心，动态绑定 (Type Registry)
      - **优点**：高可修改性、可变性、集成性
      - **缺点**：(相对) 性能风险? (但在案例分析中被评为高性能因为组件少)
    - **Banking架构**：应用逻辑嵌入框架组件，Main模块绑定
      - **缺点**：低可修改性、低安全性 (应用信息暴露)、低可重用性
  - **分析过程**
    - **效用树**：确定高优先级场景 (如：错误处理-可靠性, 快速处理-性能)
    - **风险/非风险识别**
      - **Hoover**：安全性(非风险), 性能(非风险), 可修改性(强)
      - **Banking**：安全性(风险), 性能(风险), 可修改性(弱)
    - **敏感点/权衡点**
      - **权衡点**：应用程序嵌入系统的位置数量 (影响可修改性 vs ?)
